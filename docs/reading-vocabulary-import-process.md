# 閱讀文章單字匯入流程規則

## 1. 目前狀態摘要

* 閱讀功能已完成收尾檢查。
* Batch 01 已正式匯入並完成匯入後單字功能檢查。
* Batch 02 已正式匯入並完成匯入後單字功能檢查。
* 目前正式日文單字庫總數為 3145。
* Batch 01 匯入 19 個閱讀單字。
* Batch 02 匯入 26 個閱讀單字。
* 目前閱讀單字盤點狀態可參考 `docs/reading-vocabulary-audit.md`。
* 匯入紀錄可參考 `docs/reading-vocabulary-import-log.md`。

## 2. 方案 C 的基本原則

1. 每一批都必須先整理候選清單，不得直接匯入。
2. 每一批候選詞都必須人工確認 `[x]` 後才能正式匯入。
3. 正式匯入時只能匯入 `[x]` 詞。
4. `[ ]` 未勾選詞不得匯入。
5. 標記「需確認」「疑似重複」「暫緩加入」的詞不得匯入。
6. 每次正式匯入後必須做單字功能檢查。
7. 每次正式匯入後必須更新匯入紀錄。
8. 每次正式匯入後若單字庫數量增加，必須同步更新健康檢查腳本中的單字數量。
9. 每一階段任務都應該拆開，不要一次做候選整理、勾選、匯入、檢查。
10. 不要因匯入單字而修改閱讀文章、閱讀功能或 UI/CSS。

## 3. 標準流程總覽

### Step 1：整理候選清單

目的：
從閱讀文章單字盤點資料中整理下一批候選詞。

輸出文件範例：
`docs/reading-vocabulary-import-batch-03.md`

重點：

* 只整理候選詞。
* 不修改 `vocabulary.json`。
* 不匯入單字。
* 候選詞數量建議 30～50 個。
* 預設勾選欄全部為 `[ ]`。

### Step 2：人工確認勾選

目的：
從候選清單中挑選適合正式匯入的詞。

修改文件範例：
`docs/reading-vocabulary-import-batch-03.md`

重點：

* 只把安全詞從 `[ ]` 改成 `[x]`。
* 不修改 `vocabulary.json`。
* 不匯入單字。
* 建議每批勾選約 20～30 個。
* 不勾選疑似重複、需確認、暫緩加入、太長片語或詞性不明的詞。

### Step 3：正式匯入

目的：
將已勾選 `[x]` 的詞正式加入 `vocabulary.json`。

修改檔案通常包含：

* `vocabulary.json`
* `docs/reading-vocabulary-import-log.md`
* `scripts/auditSiteHealth.js`
* 必要時更新 `docs/reading-vocabulary-audit.md`

重點：

* 只匯入 `[x]` 詞。
* 匯入前要再次檢查是否已存在於正式單字庫。
* ID 必須接續目前最大 ID。
* 新增詞必須符合既有欄位格式。
* 匯入後要更新匯入紀錄。
* 匯入後若單字總數變更，要更新健康檢查腳本。

### Step 4：匯入後單字功能檢查

目的：
確認新匯入詞不會影響搜尋、分類、程度篩選、單字練習、單字測驗。

重點：

* 檢查新增詞是否可搜尋。
* 檢查詞性分類是否正常。
* 檢查 N4 篩選是否正常。
* 檢查 N5 不應出現 N4 新詞。
* 檢查單字練習是否正常。
* 檢查單字測驗是否正常。
* 確認沒有 `undefined`、`null`、空白題目或空白選項。

## 4. 候選詞整理規則

優先選入候選清單的詞：

1. N4 常見生活詞。
2. 閱讀理解重要詞。
3. 多篇閱讀文章重複出現的詞。
4. 假名明確的詞。
5. 詞性明確的詞。
6. 中文意思明確的詞。
7. 適合初學者練習的詞。
8. 名詞、動詞、い形容詞、な形容詞、副詞、數量詞等適合單字庫收錄的詞。

暫緩或不建議選入的詞：

1. 太長的片語。
2. 完整句子。
3. 文法表現，不適合當單字。
4. 假名不確定。
5. 詞性不確定。
6. 中文意思需要人工確認。
7. 疑似與既有單字重複。
8. 太專門或只出現一次的低優先詞。
9. 需確認是否適合 N4 的詞。

## 5. 勾選 `[x]` 規則

`[ ]` 代表尚未確認，不得匯入。
`[x]` 代表已人工確認，可以在正式匯入任務中處理。

可以勾選 `[x]` 的條件：

1. 詞本身不在正式單字庫中。
2. 不是 Batch 01 或 Batch 02 已匯入詞。
3. 沒有標記「疑似重複」。
4. 沒有標記「需確認」。
5. 沒有標記「暫緩加入」。
6. 不是過長片語。
7. 不是完整句子。
8. 假名、詞性、中文意思都明確。
9. 適合 N4 或初學者閱讀理解。

不可勾選的詞：

1. 疑似重複詞。
2. 需人工確認詞。
3. 暫緩加入詞。
4. 詞性不明詞。
5. 假名不明詞。
6. 太長片語。
7. 只適合放例句或文法說明的表現。

## 6. 正式匯入規則

1. 只讀取候選清單中 `[x]` 的詞。
2. 再次比對 `vocabulary.json`，避免重複新增。
3. 若發現已存在，請不要新增。
4. 若發現疑似重複，請不要新增，並在回報中說明。
5. 新增 ID 必須接續目前最大 ID。
6. 新增資料必須符合 `vocabulary.json` 既有欄位格式。
7. 不得建立新格式。
8. 不得刪除既有單字。
9. 不得修改 Batch 01、Batch 02 已匯入詞，除非另有明確修正任務。
10. 匯入後必須更新 `docs/reading-vocabulary-import-log.md`。
11. 匯入後若單字總數增加，必須更新 `scripts/auditSiteHealth.js` 的預期數量。
12. 可視需要更新 `docs/reading-vocabulary-audit.md`，使盤點狀態反映最新單字庫狀態。

## 7. 匯入後檢查規則

每次正式匯入後必須檢查：

1. 新增詞都存在於 `vocabulary.json`。
2. ID 沒有重複。
3. `word` 欄位非空。
4. `kana` 欄位非空。
5. `meaning` 欄位非空。
6. `partOfSpeech` 欄位非空。
7. `level` 欄位正確。
8. `example`、`exampleKana`、`exampleMeaning` 格式正確。
9. 新增詞可以用漢字搜尋。
10. 新增詞可以用假名搜尋。
11. 詞性分類正常。
12. 程度篩選正常。
13. 單字練習正常。
14. 單字測驗正常。
15. 匯入紀錄與實際新增詞一致。
16. 沒有造成 `undefined`、`null`、空白資料。
17. 沒有改到閱讀文章、閱讀功能、UI/CSS 或英文網站。

## 8. 每次任務建議執行的檢查指令

```bash
node --check script.js
node --check japaneseReadingQuestions.js
node scripts/auditSiteHealth.js
node scripts/audit-reading-vocabulary.js
node scripts/check-reading-ruby.js
git diff --check
git status --short
```

補充說明：

* `scripts/check-reading-ruby.js` 可能會出現既有 reading rubyTerms warning，只要 exit code 為 0，且不是本次匯入造成，就不阻擋合併。
* 若環境沒有 Playwright 或瀏覽器自動測試設定，可以用 Node 靜態檢查與人工實機測試補足。
* 若有實際修改 `vocabulary.json`，請額外檢查單字庫總數、ID 不重複、必要欄位非空。

## 9. PR 與合併規則

1. 候選清單整理完成後，可先合併，作為下一步勾選任務依據。
2. 勾選任務完成後，應先合併，作為正式匯入任務依據。
3. 正式匯入任務完成後，檢查通過才合併。
4. 匯入後單字功能檢查若沒有修改檔案，可不用建立 PR，直接記錄完成。
5. 若匯入後檢查有必要小修，需建立 PR 並合併後才算收尾。
6. 不要在同一個任務中混合候選整理、勾選、正式匯入、匯入後檢查。

## 10. Batch 03 建議流程

1. `方案 C Batch 03 候選清單整理任務`
2. `方案 C Batch 03 人工確認勾選任務`
3. `方案 C 正式匯入任務 Batch 03`
4. `Batch 03 匯入後單字功能檢查任務`

Batch 03 應從目前 `docs/reading-vocabulary-audit.md` 與 `docs/reading-vocabulary-review-list.md` 中尚未收錄、尚未匯入、且安全性高的候選詞開始整理。

## 11. 注意事項

1. 不要一次匯入太多詞。
2. 每批建議正式匯入約 20～30 個詞。
3. 候選清單可以 30～50 個。
4. 如果候選詞品質下降，應先停止匯入，改做人工整理。
5. 若剩餘候選詞多為需確認詞，不要硬做下一批。
6. 動詞應優先收錄辭書形。
7. 複合詞性要注意分類邏輯，例如 `名詞/する動詞`、`名詞/副詞`、`名詞/な形容詞`。
8. 搜尋與測驗功能應能處理複合詞性。
9. 每批匯入後都要更新匯入紀錄。
10. 每批匯入後都要更新健康檢查數量。
