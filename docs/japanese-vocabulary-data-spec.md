# 日文單字資料規格文件

## 1. 文件目的

本文件定義 `vocabulary.json` 的日文單字資料格式與維護規則，作為後續內容品質檢查、資料修正與匯入新單字時的正式基準。

本文件目的包含：

- 定義 `vocabulary.json` 的資料格式。
- 統一日文單字維護規則。
- 作為後續內容品質檢查、資料修正、匯入新單字的標準。
- 避免每次任務重新判斷 `kana`、`exampleKana`、詞性、中文意思與例句規則。
- 確保日文單字資料能維持與現有前端顯示、分類、搜尋、練習與測驗流程相容。

## 2. 目前資料庫基準

目前日文單字資料庫基準如下：

| 項目 | 基準 |
| --- | ---: |
| 檔案 | `vocabulary.json` |
| 總筆數 | 3179 |
| N5 | 1000 |
| N4 | 2179 |

維護規則：

- 每筆資料必須保留固定 `id`。
- 不可任意重排 `id`。
- 不可任意刪除或合併單字。
- 若需要新增或刪除單字，必須另開專門任務。
- 一般內容修正任務不得順手改變總筆數、N5 / N4 分布或既有 `id` 對應關係。

## 3. 必要欄位規格

每筆單字資料必須包含以下欄位：

- `id`
- `level`
- `word`
- `kana`
- `meaning`
- `partOfSpeech`
- `example`
- `exampleKana`
- `exampleMeaning`

欄位規則：

- 每個欄位都必須存在。
- 每個欄位都不可為空字串。
- 不可出現 `undefined`。
- 不可出現 `null`。
- 不可出現格式錯亂，例如欄位值放錯欄位、JSON 結構破損或欄位名稱拼錯。
- 欄位名稱不可任意變更。
- 目前前端與檢查腳本都依賴這些欄位；若修改欄位名稱或刪除欄位，可能造成單字卡、搜尋、分類、練習與測驗顯示異常。

## 4. id 規則

`id` 是單字資料的固定識別碼，必須遵守以下規則：

- `id` 必須唯一。
- `id` 不可重複。
- 既有 `id` 不可任意變更。
- 不可因整理資料而重新排序 `id`。
- 新增單字時應延續既有最大 `id` 往後新增。
- 若移除單字，必須另行評估是否保留 `id` 空缺，不可在一般修正任務中私自重排。
- 品質檢查或修正報告應列出涉及的 `id`，以利追蹤。

## 5. level 規則

`level` 表示單字程度，目前只允許以下值：

- `N5`
- `N4`

規則：

- `level` 不可空白。
- 不可使用非標準值，例如 `n5`、`N４`、`初級`、`N3`。
- 如果未來要加入 N3，必須另開規格擴充任務。
- 若某些單字難度有爭議，應先標記 Review，不要在大型任務中批次更改。
- 閱讀場景常見但偏長、偏場景化或偏 N4 後段 / N3 的詞彙，可先保留並標記 Review，再於專門任務中評估。

## 6. word 規則

`word` 是單字卡主顯示的日文詞條。

規則：

- 動詞原則上使用辭書形。
- い形容詞使用基本形。
- な形容詞使用常見詞幹或既有資料慣例。
- 名詞使用自然詞形。
- 片假名外來語應保留片假名。
- 不應把外來語強制轉成平假名。
- 不建議收錄太長、太片語化、太商務或超出 N5～N4 太多的詞。
- 若詞條與既有資料高度同義，應考慮 `meaning`、`example` 或測驗相容性，而不是直接刪除。
- 複合詞若來自閱讀或日常場景，可保留，但應注意是否過度場景化或與既有基本詞高度重疊。

## 7. kana 規則

`kana` 是 `word` 的讀音輔助欄位。

規則：

- 漢字詞的讀音通常使用平假名。
- 片假名外來語允許在 `kana` 中保留片假名。
- 外來語不需要強制轉成平假名。
- 允許長音符號 `ー`。
- 允許促音、小音。
- 允許平假名與片假名依自然日文書寫混用。
- 若 `word` 本身是片假名外來語，`kana` 可與 `word` 相同或接近。
- 例如 `センター` 不應強制改成 `せんたー`。
- `kana` 不應殘留不必要的漢字。
- `kana` 不應出現中文、英文解釋或多餘標點。

外來語規格確認結論已納入本文件：`kana` 欄位允許保留片假名外來語；外來語不應只為了形式統一而強制轉成平假名。

## 8. meaning 規則

`meaning` 是單字的中文意思，應使用台灣繁體中文。

規則：

- 翻譯應自然、簡潔、適合初學者。
- 避免中國大陸用語。
- 避免機器翻譯感。
- 避免過度冗長。
- 避免太簡略造成誤解。
- 若日文詞有多義，應優先採用 N5～N4 與網站閱讀情境最常見的意思。
- 若有同義詞或近義詞，應避免多個詞條的 `meaning` 完全相同而造成測驗歧義。
- 需要時可用「使用、利用」這類方式降低歧義。
- 不應為了避免重複而改成不自然中文。
- 若日中同形詞可能造成誤解，應以更自然的台灣繁體中文補足語意。

## 9. partOfSpeech 規則

`partOfSpeech` 必須能被現有分類邏輯解析，詞性文字需維持既有資料慣例。

常見詞性包含：

- `名詞`
- `動詞`
- `い形容詞`
- `な形容詞`
- `副詞`
- `數詞`
- `數量詞`
- `代名詞`
- `連體詞`
- `感嘆詞`
- `名詞/する動詞`

規則：

- 複合詞性允許使用 `/`、`／`、`・`、`、`、`,` 等既有可解析分隔符號。
- 不應任意創造新詞性名稱。
- 若需要新增詞性，必須先確認 `script.js` 中分類邏輯是否支援。
- 詞性會影響分類篩選與測驗干擾選項分組，因此不可隨意修改。
- 若詞性有爭議，應標記 Review 並保留修正理由。

## 10. example 規則

`example` 是日文例句。

規則：

- 例句應自然、文法正確。
- 助詞、動詞變化、形容詞變化需正確。
- 句子長度應適合 N5～N4 學習者。
- 例句應能清楚呈現 `word` 的用法。
- 避免太商務、太抽象、太成人、太艱深或太文學化。
- 優先使用日常生活、學校、家庭、交通、購物、餐廳、天氣、旅行等情境。
- 敬體與普通體應一致。
- 不應為了展示難句而犧牲初學者可理解性。
- 若例句為閱讀衍生場景，仍應確認其可作為獨立單字卡例句理解。

## 11. exampleKana 規則

`exampleKana` 是 `example` 的讀音輔助句。

規則：

- 漢字通常轉為平假名。
- 外來語允許保留片假名。
- 不需要把外來語強制轉成平假名。
- 允許平假名與片假名混用。
- 允許長音符號 `ー`。
- 例句中的外來語若在 `example` 中為片假名，`exampleKana` 通常也應保留片假名。
- `exampleKana` 不是「完全平假名化」欄位，而是「自然日文讀音輔助」欄位。
- 不應把 `センター` 改成 `せんたー`。
- 不應漏字、多字或與 `example` 不一致。
- 不應殘留非外來語漢字。
- 助詞、動詞變化與句子結構應與 `example` 對應。

## 12. exampleMeaning 規則

`exampleMeaning` 是例句中文翻譯，應使用台灣繁體中文。

規則：

- 必須符合 `example` 的意思。
- 翻譯應自然，不要太直譯。
- 應避免中國大陸用語。
- 時間、地點、人物、動作不可漏譯。
- 若日文是報名、登記、填寫資料情境，應依語境選擇「登記」「填寫」「註冊」等自然用語。
- 不要只因為日文是 `登録` 就固定翻成「登錄」。
- 例句中文應幫助學習者理解該單字用法。
- 若中文缺少主語但自然且不影響理解，可依台灣中文習慣保留。

## 13. 外來語片假名規格

外來語片假名規格如下：

- 外來語在 `word` 中通常保留片假名。
- 外來語在 `kana` 中允許保留片假名。
- 外來語在 `exampleKana` 中允許保留片假名。
- 外來語不應強制轉成平假名。
- 這是目前資料庫既有慣例。
- id 3178 的 `センター` 符合目前規格，不需要修改。
- 未來若要變更外來語規格，必須另開全庫規格調整任務，不可在單筆修正任務中局部改動。

可作為規格示例的外來語包含：

- `センター`
- `ホテル`
- `レストラン`
- `コンビニ`
- `スーパー`
- `バス`
- `タクシー`
- `テレビ`
- `パソコン`
- `メール`
- `インターネット`

以上詞彙是外來語表記規格示例，不代表本資料庫目前一定全部收錄。

## 14. 重複與近義詞處理規則

發現重複或近義詞時，應遵守以下原則：

- 發現近義詞時，不應直接刪除。
- 應先判斷兩者是否有詞性、語氣、正式程度、使用情境差異。
- 可透過 `meaning`、`example`、`exampleMeaning` 區分。
- 若兩者會造成測驗歧義，應標記 Review 或 FixLater。
- 修正時以最小改動為原則。
- 不可為了降低重複而大量合併資料。

需特別注意的情境包含：

- `留守番電話` vs `留守電`：正式名稱與縮略語可能不同，不能直接刪除其中一筆。
- `お腹` vs `腹`：語氣、禮貌度與日常使用感不同，`meaning` 不宜完全造成測驗歧義。
- `利用` vs `利用する`：名詞 / する動詞與動詞化用法需依詞性與例句區分。
- `着く` vs `到着する`：日常語與較正式的到達表現需依語境區分。

## 15. 測驗相容性規則

單字資料會影響測驗功能，維護時需注意：

- `meaning` 會影響單字測驗選項。
- 若多個詞條 `meaning` 完全相同，可能造成測驗中兩個答案都像正解。
- `partOfSpeech` 會影響分類與干擾選項分組。
- 過長中文可能造成手機版選項擁擠。
- 過於相近的中文意思需標記 Review。
- 修正單字資料時，需確認測驗仍可正常產生選項。
- 不應為單字內容問題修改測驗邏輯，除非另開功能任務。
- 若問題源自單字資料，應優先以資料最小修正處理。

## 16. 修改任務原則

未來修改 `vocabulary.json` 時應遵守：

- 小範圍修正。
- 先檢查再修正。
- 每次修正需列出 `id`。
- 不可大量重寫。
- 不可重排 `id`。
- 不可未經確認刪除單字。
- 不可順手修改英文頁面。
- 不可順手修改日文功能程式。
- 不可修改非任務範圍檔案。
- 若檢查後發現需要修正單字資料，應只在規格文件或報告中記錄維護原則，不可在文件化任務中順手修正資料。
- 修正後需確認總筆數。
- 修正後需確認 N5 / N4 分布。
- 修正後需確認 JSON parse。
- 修正後需確認必要欄位完整。
- 若外來語規格需要改變，必須另開全庫規格調整任務。

```bash

node --check script.js
node scripts/auditSiteHealth.js

node scripts/check-japanese-vocabulary-view-split.js
node scripts/check-japanese-main-entry.js

```

若任務涉及資料筆數、必要欄位、`id` 或 level 分布，建議另以一次性 Node.js 或 Python 腳本確認：

- `vocabulary.json` 可 parse。
- 總筆數仍為 3179，除非任務明確允許新增或刪除。
- N5 / N4 分布仍為 N5 1000、N4 2179，除非任務明確允許調整。
- 必要欄位皆存在且非空。
- 沒有 `null`。
- 沒有重複 `id`。
- 沒有非預期的 level。
- 目標修正 `id` 仍存在。

## 18. 後續建議

後續日文單字資料維護建議如下：

- 可以繼續下一批日文單字內容品質檢查。
- 建議每批先做只讀品質檢查報告，再做小範圍修正任務。
- 建議優先檢查最近匯入或閱讀文章衍生的單字。
- 不建議一次檢查或修改全部 3179 筆。
- 若未來要加入 N3 或更多級別，需先擴充本規格。
- 若檢查後發現需要修正 `vocabulary.json`，應另開資料修正任務並列出目標 `id`，不得在規格文件化任務中直接修改資料。

## 20. 明確結論

1. `vocabulary.json` 目前正式基準為 3179 筆，其中 N5 1000 筆、N4 2179 筆。
2. 每筆資料必須保留固定 `id` 與九個必要欄位。
3. `kana` 與 `exampleKana` 是自然日文讀音輔助欄位，不是強制完全平假名化欄位。
4. 外來語可在 `word`、`kana`、`exampleKana` 中保留片假名。
5. id 3178 的 `センター` 符合目前資料規格，不需要修改。
6. 未來若要更改外來語或欄位規格，必須另開規格擴充或全庫調整任務。
