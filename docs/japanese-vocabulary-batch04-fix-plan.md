## 4. Batch 04 實際修改原則

- `exampleKana` 必須與 `example` 日文內容一致，且應使用全假名；不得混入漢字或漏掉助詞、標點造成 ruby / reading 對應錯誤。
- `exampleMeaning` 應自然、符合繁體中文語感，並準確對應日文例句；不得使用直譯到不自然或語義偏離的中文。
- `meaning` 不可過度簡略或造成誤解；若例句呈現特定語義，釋義應涵蓋該語義，例如「吸う」應補「抽（菸）」。
- `partOfSpeech` 應維持與單字實際用法一致；若使用「名詞/する動詞」、「名詞/副詞」、「名詞/な形容詞」等混合標示，應有資料庫策略支撐，不宜機械式更改。
- kana 修正時不可混入漢字；片假名外來語的 `kana` 欄是否使用平假名，應依既有資料策略一致處理。
- level 調整需有明確教學理由；偏書面或疑似偏難詞可先標註語域並人工確認，不應自動升降級。
- 不應刪除任何單字，除非另有明確任務指示；表記重複或近義重複項目應先提出人工確認方案。
- 對「談論單字本身」的元語言例句，應優先改為自然生活、學校、工作或旅遊情境，讓例句實際示範該詞的語法與語義。
- 實際修改 `vocabulary.json` 時，每筆應保留原 id，並只修改經人工確認的欄位；不得連帶修改 `script.js` 或現有功能。
- 實際修改後必須重新執行 JSON、JavaScript 語法、站台健康與日文單字相關檢查，並確認 `git diff --check` 無空白錯誤。
